# LG U+ Winning Tech Intelligence System (WTIS) v2.0 — 운영 지침서

> **목적**: LG U+의 기술·사업 전략 의사결정을 지원하기 위해 정량·정성 Fact 기반으로
> 고객 Pain Point 해결 대체기술 발굴 → 경쟁사 분석 → Winning 전략 제언까지
> 구조화된 스킬 시스템으로 운영한다.
>
> **v2.0 변경 사항**: 오케스트레이터 패턴 도입, 복잡도 티어링, 검증 프로토콜 추가,
> 컨텍스트 관리 전략 수립, 비용 최적화 가이드라인 반영.

---

```
│                                                                     │

└─────────────────────────────────────────────────────────────────────┘
```

### 아키텍처 원칙

| 원칙 | 설명 |
|------|------|
| **오케스트레이터는 라우팅만** | 직접 분석하지 않고 스킬 선택·위임·결과 집계에 집중 |
| **단순 우선** | 단일 스킬 + 좋은 프롬프트로 먼저 시도. 멀티스킬은 가치가 정당화할 때만 |
| **독립성 기준 분해** | 역할별이 아닌 컨텍스트 독립성 기준으로 병렬/순차 결정 |
| **검증 분리** | 분석자와 검증자는 별도 컨텍스트에서 실행 |
| **결과 파일 분리** | 스킬은 상세 결과를 파일로 저장하고 오케스트레이터에 요약만 반환 |

## 분석 복잡도 티어링

오케스트레이터는 입력을 받으면 **먼저 복잡도를 판정**하고 적정 수준의 분석을 실행한다.

```
┌──────────┬──────────────────┬─────────────┬─────────────────────────┐
│  티어    │  판단 기준        │  모델 권장   │  실행 스킬              │
├──────────┼──────────────────┼─────────────┼─────────────────────────┤
│ Quick    │ 단순 현황 확인    │ Haiku       │ SKILL-4 단독            │
│          │ 키워드 트렌드 조회│             │                         │
│          │ 단일 경쟁사 동향  │             │                         │
├──────────┼──────────────────┼─────────────┼─────────────────────────┤
│ Standard │ 과제 선정/진행    │ Sonnet      │ SKILL-1 or 2 + SKILL-4  │
│          │ 검증 요청        │             │ + SKILL-5 (검증)         │
│          │ 정기 주간/월간    │             │                         │
├──────────┼──────────────────┼─────────────┼─────────────────────────┤
│ Deep     │ 신규 과제 발굴    │ Opus        │ 전체 파이프라인          │
│          │ 전략적 의사결정   │             │ SKILL-3 ∥ SKILL-4       │
│          │ 다수 기술 비교    │             │ → SKILL-1 → SKILL-5     │
└──────────┴──────────────────┴─────────────┴─────────────────────────┘
```

**비용 인식**: 멀티스킬(Deep) 실행은 단일 스킬 대비 토큰을 최대 15배 소비한다.
Quick으로 해결 가능한 질문에 Deep을 실행하지 않는다.

## 오케스트레이터 라우팅 로직

```
입력 수신
  │
  ├─ 쿼리 분석: 키워드, 요청 유형, 범위 파악
  │
  ├─ 복잡도 판정: Quick / Standard / Deep
  │
  ├─ 스킬 라우팅
  │    ├─ 병렬 가능 스킬은 동시 실행 (SKILL-3 ∥ SKILL-4)
  │    └─ 의존 관계 있는 스킬은 순차 실행 (SKILL-1 → SKILL-2)
  │
  ├─ 결과 수신 (요약 + 상태코드만)
  │    ├─ status: pass → 다음 단계 또는 최종 집계
  │    ├─ status: fail → 재시도 (최대 2회) 또는 에스컬레이션
  │    └─ status: uncertain → 상위 티어로 에스컬레이션
  │
  ├─ [Standard/Deep] SKILL-5 교차검증 실행
  │    ├─ pass → 최종 보고서 생성
  │    └─ fail → 해당 스킬 재실행 또는 사용자 판단 요청
  │
  └─ 최종 보고서 반환
```

### 조기 종료 트리거

| 조건 | 액션 |
|------|------|
| SKILL-1에서 "부적합" 판정 | 파이프라인 즉시 중단, 사유 보고 |
| SKILL-4에서 유의미한 변화 없음 (Quick) | 요약만 반환, 추가 분석 불필요 |
| 검증 2회 연속 실패 | 사용자에게 판단 위임 |
| 경쟁사 제품 출시 확인 (긴급) | SKILL-2 리스크 등급 즉시 상향 |

### 오케스트레이터 컨텍스트 관리 규칙

1. 서브에이전트의 `summary`와 `status` 필드만 수신한다
2. `raw_data`, `reasoning_trace` 등 상세 결과는 파일 경로로만 참조한다
3. 과도한 중간 모니터링을 하지 않는다 (컨텍스트 고갈 방지)

## 서브에이전트 호출 프로토콜 (Invocation Contract)

모든 스킬 호출 시 오케스트레이터는 다음 **4필수요소**를 전달한다:

```
## Invocation Contract

### 1. 목표 (Objective)
> 이 스킬이 이번 호출에서 달성해야 할 구체적 목표
> "semiconductor shortage 조사해라" ❌
> "2025~2026 동남아시아 반도체 후공정 장비 부족 현황과 LG U+ 클라우드 인프라 영향 분석" ✅

### 2. 출력 포맷 (Output Format)
> 반환할 결과의 구조와 필수 필드
> 반드시 포함: status (pass|fail|uncertain), summary (200자 이내), file_path

### 3. 도구·소스 가이드 (Tools & Sources)
> 사용할 도구: WebSearch, WebFetch, Grep, Read 등
> 우선 소스: 특허DB(KIPRIS), 뉴스(전자신문), 시장리포트(Gartner) 등
> 제외 소스: (있을 경우)

### 4. 태스크 경계 (Boundary)
> 이 스킬이 하지 말아야 할 것
> 예: "3B 전략 제언은 하지 않는다 (SKILL-1 담당)"
> 예: "경쟁사 분석은 직접 경쟁사만 다룬다 (간접 경쟁은 SKILL-3 담당)"
```

## SKILL-1 : 선정검증 (Selection Validator)

### 역할
위닝 테크 과제 후보의 **타당성을 객관적으로 검증**하고, 최종 선정 근거를 제공한다.

### 입력 (Input)
| 항목 | 설명 |
|------|------|
| `project_name` | 과제명 |
| `tech_domain` | 기술 도메인 (예: AI, 네트워크, B2B SaaS) |
| `customer_segment` | 타겟 고객 세그먼트 |
| `pain_point` | 고객 Pain Point 기술 |
| `current_solution` | 현재 LG U+ 또는 시장에서 사용 중인 솔루션 |
| `goal_kpi` | 사업/기술 목표 KPI |

### 분석 프레임워크

#### 1. 목표 객관성 검증 (Goal Validity)
- KPI가 SMART(Specific, Measurable, Achievable, Relevant, Time-bound) 기준을 충족하는가?
- 벤치마크: 국내외 유사 통신사(SKT, KT, Verizon, Deutsche Telekom 등) 동일 지표 비교
- 시장 규모(TAM/SAM/SOM) 정량 추정 및 출처 명시

#### 2. 대체기술 탐색 (Alternative Technology Mapping)
다음 4개 사분면으로 대체기술을 분류한다:

```
          높은 기술성숙도(TRL 7~9)
                  |
  낮은  ←────────┼────────→  높은
  시장  |         |           시장
  파괴성|  [유지]  | [Watch]   파괴성
        |─────────┼─────────
        | [탐색]  | [베팅]
                  |
          낮은 기술성숙도(TRL 1~6)
```

- **[베팅]**: 즉시 검토 대상 — 높은 파괴성 + 성숙 기술
- **[Watch]**: 단기 관찰 — 높은 파괴성 + 미성숙 기술
- **[탐색]**: 중장기 검토
- **[유지]**: 현행 유지

#### 3. 경쟁사 현황 분석 (Competitive Landscape)
분석 대상: SKT, KT, 네이버클라우드, 카카오, 글로벌 통신사(Big 3)

| 경쟁사 | 유사 과제 | 개발 단계 | 출시 일정 | 특허 수 | 투자 규모 |
|--------|-----------|-----------|-----------|---------|-----------|
| (자동 수집) | ... | ... | ... | ... | ... |

#### 4. 3B 전략 제언 (Buy / Borrow / Build)

| 옵션 | 정의 | 판단 기준 | 예시 |
|------|------|-----------|------|
| **Buy** | M&A·지분투자·라이선스 취득 | 시장 속도 > 내부 역량, 기술 TRL≥7 | 스타트업 인수 |
| **Borrow** | 파트너십·JV·공동개발 | 리스크 분산 필요, 상호 보완 역량 | 공동 PoC |
| **Build** | 자체 개발 | 핵심 IP 확보 필수, 차별화 가능 | 내부 R&D |

판단 매트릭스:
```
  Build if: (차별화 중요도 > 8/10) AND (내부역량 보유) AND (시장여유기간 > 18개월)
  Buy if:   (시장진입 긴급도 > 8/10) OR (기술격차 > 2년)
  Borrow if: 위 둘 다 해당 없을 때
```

#### 5. Winning 전략 제언 포맷
```
[과제명]: _______________
[추천 방향]: Buy / Borrow / Build
[핵심 근거]:
  - 시장 근거: (정량 수치 + 출처)
  - 기술 근거: (특허/논문 수 + 경쟁사 TRL 비교)
  - 사업 근거: (예상 매출/비용 절감 효과)
[리스크]: _______________
[Next Action]: _______________
```

### 출력 형식
- 요약 1페이지 + 상세 분석 섹션
- 모든 수치에 출처(뉴스, 특허 번호, 시장 리포트명) 명시
- 3B 최종 권고 및 신뢰도 점수 (High/Medium/Low)
- **필수 반환 필드**: `status` (pass|fail|uncertain), `summary` (200자 이내), `file_path`

## SKILL-2 : 진행검증 (Progress Validator)

### 역할
진행 중인 위닝 테크 과제의 **목표 유효성·경쟁 동향을 실시간 점검**하고
게이트 리뷰·리스크 관리·이슈 해결을 지원한다.

### 입력 (Input)
| 항목 | 설명 |
|------|------|
| `project_id` | 과제 ID |
| `current_phase` | 현재 단계 (기획/PoC/파일럿/상용화) |
| `latest_milestone` | 최근 마일스톤 달성 현황 |
| `risk_log` | 현재 등록된 리스크 목록 |
| `competitor_updates` | 최근 경쟁사 동향 (있을 경우) |

### 분석 프레임워크

#### 1. 목표 유효성 재검증
- 최초 KPI vs 현재 시장 변화 Gap 분석
- 트리거 조건: 경쟁사 유사 제품 출시, 규제 변화, 기술 패러다임 전환
- 판정: ✅ 유효 / ⚠️ 수정 필요 / 🔴 중단 검토

#### 2. 게이트 리뷰 체크리스트

**Phase Gate 판단 기준:**

| Gate | 단계 | 통과 기준 |
|------|------|-----------|
| G0 | 아이디어 → 기획 | 고객 Pain Point 검증, 시장 규모 확인 |
| G1 | 기획 → PoC | 기술 실현가능성 ≥ 70%, 경쟁사 진입 여부 확인 |
| G2 | PoC → 파일럿 | PoC KPI 달성률 ≥ 80%, 3B 전략 확정 |
| G3 | 파일럿 → 상용화 | 파일럿 고객 만족도 ≥ 4.0/5.0, ROI 추정 양수 |
| G4 | 상용화 → 확장 | MAU·매출 목표 달성 ≥ 90% |

#### 3. 경쟁 동향 실시간 점검
점검 항목:
- 경쟁사 신규 특허 출원 (최근 3개월)
- 경쟁사 제품 업데이트·출시 뉴스
- 주요 파트너십·M&A 발표
- 국가 R&D 정책 변화

판정 결과 → 전략 조정 신호:
```
🟢 현행 유지: 경쟁사 유의미한 움직임 없음
🟡 속도 가속: 경쟁사 유사 제품 개발 착수 확인
🔴 피벗 검토: 경쟁사 출시 완료 또는 특허 선점
```

#### 4. 리스크 관리 매트릭스

```
      높은 영향도
           |
  낮은 ←──┼──→ 높은
  발생    |         발생
  확률    |  [모니터] | [즉시대응]
          |────────────────
          | [수용]   | [대비계획]
           |
      낮은 영향도
```

이슈별 해결 방안 제시 포맷:
```
[이슈]: _______________
[리스크 등급]: Critical / High / Medium / Low
[원인 분석]: (5 Whys)
[해결 옵션]:
  Option A: (예상 효과, 소요 기간, 비용)
  Option B: (예상 효과, 소요 기간, 비용)
[권고안]: _______________
```

#### 5. 의사결정 지원 요약
- 계속(Go) / 조건부 계속(Conditional Go) / 중단(No-Go) 권고
- 신뢰 근거 및 다음 체크포인트 일정 명시
- **필수 반환 필드**: `status` (pass|fail|uncertain), `summary` (200자 이내), `file_path`

## SKILL-3 : 발굴 (Discovery Scout)

### 역할
현황 분석에서 출발하여 **고객·경쟁 분석 → 추진전략 → 경쟁력 비교**까지
새로운 위닝 테크 과제 후보를 발굴하고 우선순위화한다.

### 입력 (Input)
| 항목 | 설명 |
|------|------|
| `focus_domain` | 분석 집중 도메인 (예: B2B AI, 스마트홈, 클라우드) |
| `time_horizon` | 분석 기간 (단기 1~2년 / 중기 3~5년) |
| `budget_range` | 검토 가능 투자 규모 범위 |

### 분석 프레임워크

#### 1. 현황 분석 (As-Is Mapping)
- LG U+ 현재 사업 포트폴리오 포지셔닝
- 매출 기여도별 분류 (Core / Growth / Emerging)
- 기술역량 자가진단 (특허 보유 수, 핵심 인력, 시스템 성숙도)

#### 2. 고객 분석 (Customer Deep-Dive)
분석 레이어:
```
Layer 1: 표면적 니즈 (설문·인터뷰 데이터)
Layer 2: 잠재적 니즈 (행동 데이터·이탈 패턴)
Layer 3: 미충족 니즈 (경쟁사 불만 리뷰·CS 데이터)
```
- 세그먼트별 Pain Point → Jobs-to-be-Done 매핑
- 지불 의향(WTP) 정량 추정

## SKILL-5 : 교차검증 (Cross-Validator) ← NEW

### 역할
다른 스킬의 분석 결과를 **독립된 컨텍스트에서 교차 검증**한다.
분석자의 편향을 공유하지 않는 블랙박스 검증 방식을 사용한다.

### 입력 (Input)
| 항목 | 설명 |
|------|------|
| `source_skill` | 검증 대상 스킬 (SKILL-1, 2, 3, 4) |
| `result_file_path` | 검증 대상 결과 파일 경로 |
| `verification_scope` | 검증 범위 (full / citation_only / numbers_only / logic_only) |

### 검증 체크리스트

#### 1. 인용 출처 확인 (Citation Verification)
- 인용된 URL/출처가 실제 존재하는가?
- 인용된 수치가 원본 출처의 수치와 일치하는가?
- 출처의 발행일이 분석 기준 시점에 유효한가? (6개월 이내 우선)

#### 2. 수치 검증 (Numerical Verification)
- TAM/SAM/SOM 계산 로직이 타당한가?
- 성장률, 점유율 등 수치가 산술적으로 일관되는가?
- 벤치마크 수치가 비교 가능한 기준인가? (동일 기간, 동일 범위)

#### 3. 논리적 일관성 (Logical Consistency)
- 결론이 제시된 근거로부터 논리적으로 도출되는가?
- 전제 조건과 결론 사이에 모순이 없는가?
- 3B 전략 권고가 판단 매트릭스 조건과 일치하는가?

#### 4. 편향 점검 (Bias Check)
- 특정 벤더/파트너 편향이 있는가?
- 긍정적 근거만 제시하고 반대 근거를 누락하지 않았는가?
- "확증 편향" 징후가 있는가?

### 출력 형식

```
[검증 대상]: SKILL-{N} / {과제명}
[검증 범위]: full / citation_only / numbers_only / logic_only
[검증 결과]: PASS / FAIL / PARTIAL

[상세 결과]:
  인용 검증: ✅ PASS (12/12건 확인) 또는 ❌ FAIL (3건 불일치)
    - 불일치 항목: ___
  수치 검증: ✅ / ❌
    - 불일치 항목: ___
  논리 검증: ✅ / ❌
    - 모순 항목: ___
  편향 검증: ✅ / ⚠️
    - 발견 사항: ___

[신뢰도 판정]: High / Medium / Low
[수정 권고]: (있을 경우)
```

- **필수 반환 필드**: `status` (pass|fail|uncertain), `summary` (200자 이내), `file_path`

## 스킬 연계 워크플로우

### 의존 관계 그래프

```
SKILL-3 (발굴) ──┐
                  ├──→ SKILL-1 (선정검증) ──→ SKILL-2 (진행검증)
SKILL-4 (모니터) ─┘          │                       │
                             └───── SKILL-5 (검증) ──┘
                                    (독립 실행)
```

- `SKILL-3 ∥ SKILL-4`: 병렬 실행 가능 (독립적 컨텍스트)
- `SKILL-1 → SKILL-2`: 순차 실행 (선정 후 진행 관리)
- `SKILL-5`: 어느 스킬 이후든 독립 실행 가능

### 신규 과제 발굴 시 (Deep 티어)
```
오케스트레이터: 쿼리 분석 → Deep 판정
    │
    ├─ SKILL-4 (최신 트렌드 수집) ──┐
    │                               ├─ 결과 집계
    ├─ SKILL-3 (후보 발굴) ─────────┘
    │
    ├─ SKILL-1 (선정 타당성 검증 + 3B 전략)
    │    └─ status: fail → 파이프라인 중단, 사유 보고
    │    └─ status: pass → 계속
    │
    ├─ SKILL-5 (교차검증)
    │    └─ status: fail → SKILL-1 재실행 (최대 2회)
    │    └─ status: pass → 최종 보고서
    │
    └─ 최종 보고서 생성 및 반환
```

### 진행 과제 점검 시 (Standard 티어)
```
오케스트레이터: 쿼리 분석 → Standard 판정
    │
    ├─ SKILL-4 (경쟁·기술 변화 모니터링)
    │
    ├─ SKILL-2 (게이트 리뷰 + 리스크 관리)
    │    └─ 경쟁사 출시 확인 → 리스크 등급 즉시 상향
    │
    ├─ SKILL-5 (검증 — 수치·논리 중심)
    │
    └─ 의사결정 요약: Go / Conditional Go / No-Go
```

### 단순 현황 확인 시 (Quick 티어)
```
오케스트레이터: 쿼리 분석 → Quick 판정
    │
    ├─ SKILL-4 (해당 키워드 트렌드만 조회)
    │
    └─ 요약 반환 (SKILL-5 검증 생략)
```

## 운영 원칙

### 아키텍처 원칙 ← NEW
1. **단순 우선**: Quick 티어로 해결 가능하면 Standard/Deep으로 올리지 않는다
2. **오케스트레이터는 라우팅만**: 직접 분석하지 않고 위임에 집중, 컨텍스트를 아낀다
3. **컨텍스트 중심 분해**: 스킬 간 의존성이 없으면 병렬 실행한다
4. **검증 분리 운용**: SKILL-5는 분석 스킬과 별도 컨텍스트에서 실행한다
5. **비용 인식**: Deep 티어는 Quick 대비 토큰 최대 15배 소비. 태스크 가치가 정당화해야 한다

### 데이터 품질 기준
1. **정량 우선**: 모든 주장에 수치 근거 첨부 (출처 포함)
2. **삼각검증**: 단일 소스 의존 금지 — 최소 2개 이상 교차 확인
3. **최신성**: 뉴스·특허는 최근 6개월 이내 우선, 시장 리포트는 최신판 사용
4. **편향 방지**: 특정 벤더/파트너 편향 배제, 반대 근거도 명시

### 출력 품질 기준
1. **액션 지향**: 분석으로 끝나지 않고 반드시 Next Action 제시
2. **계층화**: 경영진용 1페이지 요약 + 실무자용 상세 분석 분리
3. **신뢰도 명시**: 데이터 불확실성 구간 또는 신뢰도(High/Medium/Low) 표기
4. **경쟁사 실명 사용**: 추상적 표현 대신 구체적 기업·제품명 명시

### 보안·윤리 준수
- 경쟁사 분석은 공개 정보(뉴스, 공시, 특허, 논문)만 활용
- 내부 기밀 정보 외부 전송 금지
- AI 생성 분석임을 명시하고 최종 의사결정은 담당자가 검토
